## 一、痛点卡片

### 痛点1：DataTalks.Club 生产环境被 AI 一键抹除

- **场景**：2026 年，在线教育平台 DataTalks.Club 的研发环境中，开发者对 AI 编码工具 Claude Code 进行了盲目授权。由于缺乏沙箱隔离，且底层 AWS 未开启删除保护，Claude Code 机械地执行了 `terraform destroy` 指令。
- **黄金开头**：短短几秒内，两届学生、近 200 万行核心数据瞬间清零。
- **黄金钩子**：这不是一次"AI 失控"——Claude Code 只是忠实地执行了一段看似合理的代码。真正的问题是：你的系统里，危险操作有二次确认吗？基础设施有刚性红线吗？

### 痛点2：开发者沦为"人肉缝合怪"

- **场景**：深度使用 AI 编码的开发者，在单次"提示-响应"模式下反复循环：把代码贴进对话框、复制 AI 的输出、跑一遍代码、看到报错、把报错粘回去、改 Prompt、再来一遍。
- **黄金开头**：无论 Prompt 写得多巧妙，你仍然是那个把碎片缝在一起的"人肉中转站"。
- **黄金钩子**：当你被卡在这个复制粘贴循环里时，AI 并没有真正替你工作——你只是换了一种更累的方式，亲手维持着系统的运转。

### 痛点3：成百上千条 Prompt 模板一夜集体报废

- **场景**：一个应用规模稍大的团队，花了数月精心打磨了成百上千条 Prompt 模板。底层模型从 GPT-4 换成新版或开源模型后，这些模板集体失效，甚至准确率反向退化。
- **黄金开头**：你精心维护的"Prompt 圣经"，在新模型面前变成了一堆废纸。
- **黄金钩子**：每一条模板都是过去投入的真金白银，但当它们"越堆越多、越滚越重"地变成技术债时，你还能承受几次从头再来的代价？

### 痛点4：一个空格击穿全部缓存，成本瞬间翻倍

- **场景**：开发者为了享受 Prompt Caching 的 90% 成本折扣，精心设计了上下文结构。但他们在系统提示词开头插入了一个动态变量（比如"今天是 2026 年 6 月 29 日"），仅因日期变了，缓存从前缀第一个字节起全部失效。
- **黄金开头**：你只改了一个空格，整段缓存在那一瞬间集体作废，系统退回全额计费的冷启动。
- **黄金钩子**：你享受过的每一分缓存红利，都建立在一个反直觉的铁律之上——前缀必须一字不差。你的上下文排列顺序，经得起这条铁律的检验吗？

### 痛点5：AI 深夜死循环烧光 Token，无人叫醒你

- **场景**：一个开发者配置好了 AI 自动循环，让它夜间无人值守运行。AI 在循环中连续失败 50 次，没有熔断器跳闸，没有看门狗杀进程，系统静默地把 Token 额度全部烧完。
- **黄金开头**：第二天早上你醒来看账单，发现 AI 加了一整夜班——什么都没做成，只留下一条烧穿预算的消费记录。
- **黄金钩子**：当你的 Loop 在凌晨 3 点陷入自旋死循环时，你的系统是会拉响警报把你叫醒，还是安静地看着它把钱烧光？

## 三、可落地方法

### 方法1：提示词缓存分层排列（Context 装配优化）

- **适用场景**：使用 Claude、GPT 等支持提示词缓存的 API 构建 Agent 应用，需要降低 Token 成本并提升响应速度。
- **步骤**：
  1. 将每次请求携带的内容拆分为四层——（A）工具定义（最稳定）；（B）系统提示（变动频率低）；（C）历史对话（逐步增长）；（D）动态消息（每次请求都变）。
  2. 按"从静到动"顺序组装请求体，严格按 A → B → C → D 排列。
  3. 动态全局变量绝不插入系统提示开头，必须追加在对话流最末端。
  4. 检查每次请求的前缀部分（A 层 + B 层）是否逐字符一致。
  5. 计算缓存经济性：同一前缀调用次数 N > 3 即可产生净收益（首次全价，后续命中约 20% 计费）。
- **验证标准**：缓存命中率 > 80%；命中缓存的请求输入 Token 费用降至首次的 20%（±5%）。

### 方法2：分层拦截流水线（Harness 漏斗模型）

- **适用场景**：构建代码审查 Agent、CI/CD 自动化流水线。
- **步骤**：
  1. 部署确定性规则层（第一层）：集成 Semgrep 或等效静态分析工具，配置安全红线规则集，毫秒级执行，零 Token 消耗。
  2. 搭建策略网关层（第二层）：使用 OPA 对 IaC 配置设刚性阻断规则。
  3. 接入 AI 审查层（第三层）：仅通过前两层后，将代码 diff 送入 LLM 审查，注入项目上下文。
  4. 配置人类终审层（第四层）：仅将涉及架构决策、安全策略变更等核心问题路由给人工 Review。
- **验证标准**：规则层拦截率 ≥ 80%；AI 审查层误报率 < 20%；人类终审工作量下降至原来的 5% 以下。

### 方法3：循环契约六维度配置（Loop Contract）

- **适用场景**：将一次性 Agent 任务升级为可无人值守运行的自主循环。
- **步骤**：
  1. 定义 TRIGGER（触发条件）：Cron 表达式、GitHub Actions 事件、Webhook 回调或 `/loop` 命令。
  2. 划定 SCOPE（作用范围）：用文件路径白名单限定 Agent 可修改范围。
  3. 声明 ACTION（允许操作）：枚举只读、草稿写入、外部写入三档权限，按风险动态匹配。
  4. 设定 BUDGET（四维预算）：步数（max 20）、墙上时间（max 30min）、Token（max 100K）、成本（max $2）。
  5. 定义 STOP（终止条件）：CI 全绿 + 单元测试通过 + Lint 零告警 + 验收用例全部 PASS。
  6. 输出 REPORT：结构化记录每次循环的产出、检查结果、消耗和熔断状态。
- **验证标准**：10 次循环中至少 8 次在 BUDGET 内正常终止；STOP 条件满足后 Agent 确实停止。

### 方法4：熔断器 + 看门狗双重安全约束

- **适用场景**：任何以无人值守模式运行的 Agent 循环，防止死循环、资源耗尽、进程卡死。
- **步骤**：
  1. 配置熔断器：设置 `max_consecutive_failures`（3-5 次）和 `max_runtime_min`（15-60 分钟），触达即强制中断并写入 `CIRCUIT_OPEN` 状态文件。
  2. 部署看门狗：独立外部进程每 5-10 秒采样 Agent 进程 CPU 和 I/O 状态，CPU > 90% 且 I/O 为零超过 60 秒则发送 SIGKILL。
  3. 熔断恢复：Watchdog 写结构化日志并通过 Webhook 通知开发者；人工修复后删除 `CIRCUIT_OPEN` 以复位。
  4. 所有无人值守循环必须在 Worktree 或 Sandbox 中运行。
- **验证标准**：故意构造死循环确认熔断器自动中断；模拟 CPU 满载确认看门狗 60 秒内 SIGKILL；主工作区文件不受污染。

### 方法5：Maker/Verifier 双 Sub-agent 拆分

- **适用场景**：代码生成或内容创作任务，解决单 Agent "既当运动员又当裁判"的问题。
- **步骤**：
  1. 定义 Maker Agent：只赋予写入权限，聚焦"产出"，不可自行判定"已完成"。
  2. 定义 Verifier Agent：只赋予读取权限，聚焦"挑错"，持有终止权。
  3. 建立通信协议：Verifier 输出 JSON 格式检查结果，Maker 逐条认领修复。
  4. 设置轮次上限：Maker-Verifier 来回最多 5 轮，超过则强制终止并标记人工介入。
  5. 沉淀为 Skill：将两个 Agent 的 Prompt 分别固化为 `skill-maker.md` 和 `skill-verifier.md`。
- **验证标准**：双 Agent 模式一次通过率显著高于单 Agent；来回轮次中位数 ≤ 3 轮，超过 5 轮任务数为 0。

## 四、思维导图

- **Prompt to Loop 进化：从执行者到设计者的范式迁移**
  - **Prompt Engineering：沟通艺术**
    - 正确方法论（定义问题 → demonstration set → 候选 prompt → 实测 → 权衡 → 迭代）
    - 常见误区与突破（Blind Prompting / DSPy & APE 声明式框架）
    - 天花板限制（上下文窗口、缺乏记忆与工具调用、容错率低、技术债）
  - **Context Engineering：对话管理**
    - 三套方法论（MVC 轻量化装配 / GraphRAG / Just-in-Time 即时检索）
    - 三类故障模式（信息匮乏 / 信息过载 / 上下文腐烂）
    - Prompt Caching 缓存机制（成本降 90% / 前缀匹配不变性铁律 / 分层排列策略 / N>3 净收益）
  - **Harness Engineering：系统约束**
    - 核心范式与四大支柱（Agent = Model + Harness）
    - 信任边界与安全（五层防御 / 分层拦截流水线：Semgrep 80% → OPA 15% → AI → 人类 5%）
    - 设计思维与原则（八条非妥协原则 / Skill Issue 框架）
    - 实证案例（DataTalks.Club 删库 / Terminal Bench 2.0 排名 30→前5）
  - **Loop Engineering：自主迭代**
    - 核心定义与哲学（Loop = Cron + 决策器 / 机制与策略分离）
    - 三档成熟度（Open Loop → Closed Loop → Review Loop）
    - 基础设施组件（五件套+记忆 / 循环协议六维度 / 熔断器+看门狗）
    - 标准闭环与范式嵌套（Prompt ⊂ Context ⊂ Harness ⊂ Loop）
  - **开发者行动指南：从执行者到 Loop Designer**
    - 三大职责（定义终止边界 / 维护工具链 / 设计安全断路器）
    - 七天复现计划（AGENTS.md → SKILL.md → Hook → Ralph 循环 → Maker/Verifier 分离 → Loop Contract → 无人值守）
    - 终局判断（上半场拼语言艺术 / 下半场拼系统工程能力）
