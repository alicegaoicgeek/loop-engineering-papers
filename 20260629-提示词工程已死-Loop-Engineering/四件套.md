## 一、痛点卡片

### 痛点1：初级开发者/独立程序员
- **场景**：试图通过简单的对话式提示词让AI Agent自主完成复杂的代码重构或功能开发，却陷入无休止的上下文重置和手动干预中。
- **黄金开头**：你以为把Bug丢给AI就能下班？其实你正在为它的“健忘”和“自负”买单。
- **黄金钩子**：为什么你写的每一个提示词，都在被迫重复解释那些昨天才确认过的项目规范？

### 痛点2：技术团队Leader/架构师
- **场景**：团队引入AI Agent并行处理任务时，因缺乏隔离机制导致多个Agent修改同一文件，引发大量代码冲突和合并失败。
- **黄金开头**：AI没让你变快，反而让你的代码库变成了一团无法合并的乱麻。
- **黄金钩子**：当三个Agent同时声称自己解决了同一个Bug，你该如何判断谁在撒谎，谁在破坏？

### 痛点3：内容运营/研究人员
- **场景**：需要每日从海量新闻源中筛选有效选题，人工阅读效率低下且容易遗漏关键信息，导致选题滞后或质量参差不齐。
- **黄金开头**：每天花两小时刷新闻，却只产出三个平庸的选题？你的时间被低效流程偷走了。
- **黄金钩子**：如果有一个系统能自动标记出“有争议”、“缺数据”的线索，你还有精力去纠结那些显而易见的热点吗？

### 痛点4：资深工程师/代码维护者
- **场景**：长期依赖AI生成代码，逐渐丧失对底层逻辑的理解，面对AI生成的复杂堆砌代码时，既不敢轻易删除，也无法自信地Review。
- **黄金开头**：你按下了启动键，却再也看不懂引擎里发生了什么。
- **黄金钩子**：当AI替你写了80%的代码，你是否已经失去了识别那20%致命错误的直觉？

## 三、可落地方法

### 方法1：基于隔离工作区的多Agent并行开发Loop
- **适用场景**：需要同时修改多个文件或模块的代码重构、Bug修复或功能开发，避免Agent间代码冲突。
- **步骤**：
  1. 使用Git Worktree或类似工具为每个AI Agent创建独立的代码工作区（Worktree），确保物理隔离。
  2. 配置调度器，将大任务拆解并分发给不同的Agent，指定各自负责的模块或文件范围。
  3. 各Agent在独立工作区内完成编码后，向主分支提交PR或变更集。
  4. 在主分支进行合并操作，解决可能的合并冲突（如有）。
- **验证标准**：所有Agent的修改均成功合并至主分支，且通过CI流水线的全量回归测试，无代码冲突导致的构建失败。

### 方法2：构建项目技能库（Skill）的标准知识库
- **适用场景**：AI Agent频繁出现“忘记”项目规范、目录结构或历史踩坑经验的场景，用于固化长期规则。
- **步骤**：
  1. 梳理项目的关键元数据，包括：启动命令、核心目录结构、禁止触碰的文件/路径、命名规范、常见技术债及历史Bug解决方案。
  2. 将上述信息整理为结构化的Markdown文档或JSON配置文件，存入项目专用的`SKILLS.md`或知识库目录。
  3. 在Loop的初始化Prompt中，强制要求Agent读取并加载该技能库文件，作为上下文的一部分。
  4. 定期由人工审核技能库内容，更新过时规范。
- **验证标准**：Agent在执行任务时，能准确引用项目特定的命名规范或避坑指南，且在连续多次迭代中无需重复交代基础背景信息。

### 方法3：执行与验证分离的双Agent审查Loop
- **适用场景**：代码生成质量不稳定、自我评估偏差大，需要提高输出准确性的自动化测试或代码生成任务。
- **步骤**：
  1. 设计两个角色：Agent A为“执行者”，负责根据需求编写代码或内容；Agent B为“审查者”，负责独立验证结果。
  2. Agent A完成任务后，将产物提交给Agent B，严禁Agent A自行验证。
  3. Agent B依据预设的检查清单（Checklist）或运行测试脚本进行验收，若失败则生成具体的错误报告返回给Agent A。
  4. Agent A根据错误报告进行修改，直至Agent B通过验证或达到最大重试次数。
- **验证标准**：最终交付物通过所有单元测试/集成测试，且审查日志显示Agent B独立发现了至少一处Agent A初始版本中的逻辑错误或风格违规。

### 方法4：持久化状态管理的Loop记忆机制
- **适用场景**：长周期、多轮次的自动化任务（如每日代码扫描、持续的内容选题），防止Agent“失忆”导致重复工作或遗忘进度。
- **步骤**：
  1. 在外部存储（如Git仓库、数据库或云端笔记）创建一个状态记录文件（如`LOOP_STATE.md`或数据库表）。
  2. 定义状态字段，包括：已完成事项、已验证结论、被拒绝的方案、当前待处理项、上次运行时间戳。
  3. Loop启动时，首先读取该状态文件，将其作为System Prompt的一部分注入，让Agent知晓历史上下文。
  4. Loop结束时，Agent必须更新该状态文件，记录本轮成果及下一步计划，再退出。
- **验证标准**：Loop重启后，Agent能准确列出上一轮遗留的未决事项，且未重复处理已标记为“已完成”或“已否决”的任务。

### 方法5：外部系统触发的自动化工作流Loop
- **适用场景**：需要AI主动介入业务流程（如自动回复Issue、监控CI失败、生成日报），而非被动等待用户指令。
- **步骤**：
  1. 确定触发源：配置Webhook监听外部事件（如GitHub Issue标签变化、CI Pipeline状态变更、定时Cron Job）。
  2. 建立API连接：Loop程序需具备读写外部系统的能力（如通过GitHub API获取Issue详情，通过Slack API发送通知）。
  3. 设定执行逻辑：当触发事件发生时，Loop自动抓取相关数据，调用LLM进行分析或生成响应。
  4. 闭环反馈：将LLM生成的结果（如代码补丁、回复文案）直接通过API写回外部系统，并记录操作日志。
- **验证标准**：外部事件发生后，Loop能在设定延迟内（如5分钟）自动完成响应，且操作记录在外部系统中清晰可查，无需人工手动启动。

## 四、思维导图

# 提示词工程已死，Loop Engineering来了！

- **核心概念与演进**
    - 从单次提示词到自动化闭环
    - 角色转变：从执行者变为调度者
    - 核心定义：设计自动运转的小系统

- **Loop 的六大关键组件**
    - 调度机制：定时或事件触发启动
    - 并行隔离：Worktree避免冲突
    - 技能固化：将项目知识外置
    - 外部连接：打通Issue/CI等工具链
    - 独立验证：执行与审查分离
    - 持久记忆：跨轮次状态记录

- **成本考量与适用边界**
    - Token消耗：反复试错导致的高成本
    - 责任归属：AI无法替代人类担责
    - 认知风险：过度依赖导致理解力下降
    - 最佳场景：高重复、流程稳定、可验证的任务

- **典型应用场景**
    - 软件开发：自动化测试与代码修复
    - 内容运营：新闻源扫描与选题生成
    - 数据分析：定期报告与异常监控
    - 客户服务：工单自动处理与分流
