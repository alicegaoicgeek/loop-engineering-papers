## 一、痛点卡片

### 痛点1：信息滞后与遗漏
- **场景**：关注AI Agent方向的运营人员希望持续追踪同领域竞品公众号（如Datawhale）的新文章更新，而非被动依赖微信推荐。
- **黄金开头**：你是不是经常因为漏刷朋友圈或推荐算法的延迟，而错过了某个关键竞品的首发深度好文？
- **黄金钩子**：当别人还在靠记忆和手动刷新时，你是否已经建立了“有更新即刻知晓”的自动化情报网，从而抢占信息先机？

### 痛点2：脚本静默失败导致的数据黑洞
- **场景**：使用Python cron脚本抓取微信公众号文章时，因API所需的cookie token过期（通常3天有效），脚本静默报错退出，导致数据中断且无人知晓。
- **黄金开头**：最可怕的不是脚本报错，而是报错后它一声不吭地停止工作，而你第二天才发现整整两天的数据“蒸发”了。
- **黄金钩子**：你是否厌倦了在数据断更后才去排查那些隐藏在日志深处的Traceback，却早已失去了补救的最佳时机？

### 痛点3：无效执行造成的资源浪费
- **场景**：传统的定时脚本无论是否有新内容，每次触发都会执行完整的全量抓取逻辑，而在大部分“无更新”时段，这种重复劳动毫无意义。
- **黄金开头**：如果每次检查都没有新东西，为什么还要让服务器全速运转去跑一遍复杂的抓取流程？
- **黄金钩子**：在追求效率的今天，你是否意识到，那些在“无更新”时依然盲目执行的代码，正在悄悄吞噬你宝贵的Token预算和计算资源？

### 痛点4：缺乏判断力的僵化异常处理
- **场景**：当API返回“token失效”等基础设施错误时，普通脚本往往直接崩溃或无限重试，无法区分这是“内容层面需重试”还是“基础设施需人工介入”。
- **黄金开头**：面对Token失效，你是选择让程序盲目重试直到被封禁，还是能像人一样先判断状态再决定下一步？
- **黄金钩子**：当系统遇到非预期错误时，你的Agent是只会机械地抛出异常，还是具备“控制器判断力”，能自动降级处理并在恢复后无缝衔接？

## 三、可落地方法

### 方法1：基于状态文件的定时同步与异常降级机制
- **适用场景**：需要定时执行任务且依赖外部API（如Token易过期）的场景，避免无效重试和资源浪费。
- **步骤**：
  1. 在终端初始化项目，安装必要工具（如 `wx-mp-rss-core`），完成首次登录并保存凭证（如 `wx_token.json`）。
  2. 在 Claude Code 中输入 `/loop [频率] [角色定义]` 启动循环，例如 `/loop 30m 你是一个公众号文章同步 agent。工作目录是当前目录。`。
  3. 编写 Prompt 逻辑：Agent 每次执行时首先读取本地状态文件（如 `sync_state.json`）。
  4. 若状态为 `success` 且无新数据，仅更新检查次数和时间戳，直接退出以节省 Token。
  5. 若检测到 API 返回错误码（如 `ret=200003`），Agent 判断为基础设施问题（Token失效），将状态文件标记为 `token_expired` 并通知用户，下一轮触发时优先执行轻量级 Token 恢复检测，而非盲目重试业务逻辑。
- **验证标准**：
  - 正常运行时，无新数据触发下单次运行 Token 消耗极低，且状态文件正确递增 `check_count`。
  - Token 失效后，Agent 不再发送业务请求，而是输出明确的状态警告，并在 Token 恢复后自动恢复正常同步流程。

### 方法2：Loop Prompt 中的决策逻辑设计法
- **适用场景**：设计具备自主判断能力的 AI Agent 循环，需区分“业务失败”与“系统故障”。
- **步骤**：
  1. 明确定义 Agent 的“控制器”职责，即在 Prompt 中嵌入判断逻辑而非仅执行动作。
  2. 在 Prompt 中规定不同的错误响应对应不同的处理策略：
     - 内容层面错误（如数据格式不对）：记录日志并重试。
     - 基础设施层面错误（如连接超时、认证失败）：暂停执行，标记状态，等待人工介入或自动修复。
  3. 利用状态文件（JSON）作为 Agent 的长期记忆，确保多轮循环间状态的一致性。
  4. 测试不同故障场景，观察 Agent 是否能根据预设逻辑选择正确的降级路径。
- **验证标准**：
  - Agent 能在遇到非预期错误时，依据 Prompt 指令做出合理决策（如区分重试与暂停）。
  - 状态文件准确反映当前循环的健康状况（Success/Failed/Expired）。

### 方法3：先 Skill 后 Loop 的开发顺序
- **适用场景**：构建稳定的自动化工作流，确保核心功能可靠后再添加调度机制。
- **步骤**：
  1. 首先剥离调度逻辑，单独开发解决核心问题的 Skill（如：抓取文章、解析数据、存储结果）。
  2. 验证该 Skill 能独立处理正常流程和常见异常，确保其鲁棒性。
  3. 将已验证的 Skill 逻辑封装进 Claude Code 的 Prompt 中。
  4. 最后使用 `/loop` 命令赋予其定时执行能力，重点调试循环间的状态管理和异常恢复机制。
- **验证标准**：
  - Skill 在无循环环境下能一次性成功执行目标任务。
  - 加入 `/loop` 后，连续多次执行过程中，核心业务逻辑不受调度干扰，异常处理依然有效。

## 四、思维导图

# Claude Code /loop 实操教程

- **Agent 自动化触发机制**
    - `/loop` 定时重复执行
    - `/goal` 持续工作直至达标
    - `hooks` 事件确定性地触发脚本
    - 多 Agent 并行与协作

- **实战场景与痛点分析**
    - 监控竞品公众号文章更新
    - 传统 Cron 脚本的局限性
    - Token 过期导致静默失败
    - Loop 的核心优势：自主判断与状态管理

- **Loop 设计与运行流程**
    - 初始化 Prompt 与调度设置
    - 正常同步：状态读取与数据更新
    - 无更新响应：极简输出以节省 Token
    - 异常处理：Token 失效后的降级策略

- **底层原理与核心洞察**
    - 技术本质：Cron 调度 + Prompt 注入
    - 智能来源：决策逻辑完全依赖 Prompt 设计
    - 关键能力：区分内容问题与基础设施问题
    - 开发前置条件：先具备解决问题的 Skills
